DuckDB is an embedded OLAP engineA database optimized for analyzing large datasets rather than processing small transactions — optimized for scanning and filtering large datasets, not transactional workloads. Key advantages:

  • Out-of-coreCan process data larger than available RAM by spilling to disk: Spills to disk when data exceeds RAM. Handles datasets far larger than memory.
  • Columnar storage: Filter predicates are pushed down to the scan layer (filters are applied at the data-reading level so unnecessary data is never loaded). Only relevant columns are read.
  • Native CSV read: read_csv parses CSV in C++, orders of magnitude faster than Python-side parsing.
  • No server: Runs in-process, zero configuration.

SQLite is row-oriented (OLTP) — great for transactions, poor for analytical scans. PostgreSQL would work but adds operational complexity for a service that runs on a single machine.

Upload streams to a temp file in 1 MB chunks (never holds full CSV in memory).

Header validation via csv.reader (just reads first line).

DuckDB's read_csv ingests the file natively — TRY_CASTAttempts type conversion, returns NULL instead of failing on bad data handles dirty data (non-numeric years, etc.).

Data goes into movies_staging table.

Atomic swap: BEGIN; DROP movies; RENAME staging → movies; COMMIT;

Temp file deleted.

Safety net: If ingestion fails at any point, the staging table is dropped and the original movies table is untouched.

Deliberate restraint. The task is CSV → database → query. Adding a 200+ MB dependency for something DuckDB handles natively would be over-engineering. Python's csv.reader is used only for header validation (one line). DuckDB does the heavy lifting.

The first version used Python's csv.reader to iterate row-by-row, parse types manually, accumulate 50K-row batches, and call executemanyA database method that runs the same SQL statement repeatedly with different values, batching the work. This round-trips every row through Python.

The optimized version uses DuckDB's read_csv — the CSV is parsed in DuckDB's C++ engine, never touching Python. TRY_CAST replaces manual type parsing. Same correctness, 170x faster.

The upload endpoint returns 202 Accepted with a task_id. Progress is tracked in a Python dict (GIL-safePython's threading model guarantees simple operations like updating a dictionary won't cause data corruption for single-process). The SSEServer-Sent Events — a standard HTTP mechanism for the server to push updates to the browser endpoint (/tasks/{id}/events) polls this dict and emits events when state changes.

With the DuckDB native reader, ingestion is sub-second for the test dataset, so progress goes 0 → 100 almost instantly. For truly massive files (GB+), the DuckDB operation would take longer and the SSE mechanism still reports the transition. In a production system with longer ingestion, you'd add more granular progress hooks.

DuckDB is single-writerOnly one process can write to the database at a time; concurrent writes will block or fail. If two uploads arrive simultaneously, the second would fail to acquire the write lock. In production:

  • Use a task queue (Celery/DramatiqPython libraries for running background jobs across multiple workers, typically paired with a message broker like Redis) to serialize ingestion jobs
  • Return 409 Conflict if an ingestion is already in progress
  • Or queue uploads and process sequentially

Correct. This is a conscious trade-off for a single-process take-home. In production:

  • Task state → RedisAn in-memory data store commonly used for caching, session storage, and job queues or database-backed store
  • Job execution → Celery/Dramatiq with a message broker
  • Horizontal scaling → multiple workers sharing external state

DuckDB handles this out of the box — out-of-core processing spills to disk. But the API layer would need changes:

  • Pagination: Already implemented with limit/offset params and X-Total-Count header. For cursor-based pagination at extreme scale, switch to keyset paginationA more efficient way to page through large datasets using the last seen value instead of counting rows to skip.
  • Streaming responses: For very large result sets, stream JSON lines instead of building a full list.
  • Ingestion: DuckDB's read_csv scales linearly. Even at 100M rows it would be minutes, not hours.
  • Download: Already streaming — no memory concerns.

Progress reporting is unidirectional (server → client). SSE is simpler:

  • No upgrade handshake
  • Standard HTTP
  • Native browser support via EventSource APIA built-in browser feature for receiving server-pushed updates over a persistent HTTP connection
  • Auto-reconnection built in

WebSockets would be overkill for one-way status updates.

DuckDB is an OLAP engine where denormalized data is idiomatic. ILIKECase-insensitive pattern matching in SQL — like LIKE but ignores uppercase/lowercase differences (case-insensitive) is pushed down to the scan layer and runs in C++. A junction tableA separate table that maps relationships (e.g., movies-to-genres), common in normalized relational databases would add join complexity with no performance benefit — the filtered query returns 8,377 rows in 0.03s.

Trade-off: Substring matching could produce false positives ("Drama" matching "Melodrama"), but the dataset's genre vocabulary doesn't have such conflicts. If it did, you'd split on commas and use list_contains.

The CSV has entries like "III" in the year column and irregular quoting ("""Giliap"""). DuckDB's TRY_CASTAttempts type conversion, returns NULL instead of crashing on bad data converts unparseable values to NULL instead of failing. strict_mode=false tolerates quoting issues. This matches the behavior of the original Python-side parsing (which had try/except blocks) but runs in C++.
