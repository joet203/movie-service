Architecture

Overview

Uploads are streamed to disk in chunks to avoid buffering the full file in application memory. Querying and storage are handled with DuckDB because it supports efficient analytical filtering on columnar data and can operate out-of-core on datasets larger than available RAM.

Columnar storage means data is stored by column rather than by row, which is dramatically faster when a query only needs certain fields. Out-of-core processing means the database can handle data bigger than your computer's RAM by temporarily spilling to disk.

Long-running ingestion is decoupled from the request/response cycle using background task execution plus in-memory task status tracking. The endpoint returns 202 Accepted immediately with a task ID. Progress updates are exposed via Server-Sent Events (SSE).

SSE is a way for the server to push real-time updates to the browser over a standard HTTP connection. Unlike WebSockets, it's one-directional (server to client) and requires no special handshake.

Data replacement uses a staging table pattern — new data is ingested into movies_staging, then atomically swapped into movies via a transaction. This ensures queries always see a complete dataset, never partial ingestion state.

A staging table is a temporary holding area for new data. By loading data there first, the live table is never in an incomplete state. Once the staging table is ready, it replaces the live table in a single atomic operation.

Large downloads are returned with StreamingResponse over a DuckDB-generated gzip file, so the response is streamed incrementally in 64 KB chunks rather than materialized in memory.

Production note: For this take-home, in-process task state (a Python dict) was chosen for simplicity. In production, job execution and status state would be externalized (e.g., Redis, a task queue) to support horizontal scaling and persistence across restarts.

Database

Why DuckDB?

DuckDB is an embedded OLAP database that runs in-process — no separate server needed. Unlike transactional databases (PostgreSQL, MySQL) which are optimized for many small reads and writes, OLAP engines are designed for scanning and filtering large volumes of data efficiently.

OLAP stands for Online Analytical Processing. It's a category of database optimized for reading and analyzing large amounts of data (aggregations, filters, scans), rather than for frequent small updates like inserting one row at a time.

Out-of-Core Processing

Automatically spills to disk when data exceeds available RAM. Handles datasets far larger than memory without any configuration.

This means the database can process a 50 GB file on a machine with only 8 GB of RAM. It keeps active data in memory but transparently pages the rest to temporary files on disk.

Columnar Storage

Analytical queries (filters, scans) are inherently faster with columnar layout. DuckDB pushes filter predicates down to the scan layer.

Predicate pushdown means the database applies your WHERE filters at the lowest level of data reading, so it never loads rows it doesn't need. Instead of reading all 367K rows and then filtering, it skips irrelevant data blocks entirely.

ACID Transactions

Enables the atomic staging table swap during ingestion — queries never see partial data.

ACID (Atomicity, Consistency, Isolation, Durability) guarantees that data operations are all-or-nothing. If the swap fails halfway through, the database rolls back to the previous state automatically, preventing partial or corrupt data.

Native Gzip Export

COPY ... TO ... COMPRESSION 'gzip' handles CSV formatting and compression in a single C++ pass — no Python-side serialization or compression overhead.

CSV Ingestion Strategy

CSV ingestion uses DuckDB's built-in read_csv function rather than Python-side parsing. This was an intentional optimization — an initial implementation using Python's csv.reader with executemany batching was dramatically slower.

How It Works

  • TRY_CAST handles dirty data gracefully — e.g., "III" in the year column becomes NULL instead of failing. TRY_CAST attempts to convert data to a specific type (like text to integer), returning NULL instead of throwing an error if the conversion fails. This prevents a single bad row from crashing the entire import.
  • strict_mode=false tolerates quoting irregularities in the CSV. This tells DuckDB to be lenient with formatting issues in the CSV (like mismatched quotes or inconsistent escaping) rather than rejecting the file entirely.
  • Header validation is still performed via a quick csv.reader first-line check before DuckDB ingests.
  • The staging table pattern is preserved for atomic replacement.

Python's csv.reader is still used for header validation — stdlib for correctness, DuckDB for performance.

Efficiency

Memory Management

Every operation avoids loading full datasets into application memory:

Upload

UploadFile streamed to a temp file in 1 MB chunks. The entire CSV never lives in Python memory at once.

Ingestion

DuckDB reads the CSV directly from disk in its C++ engine. No Python-side row iteration means no per-row object allocation overhead.

Query

DuckDB pushes WHERE filters to the scan layer via predicate pushdown. Only matching rows are materialized into Python objects.

Predicate pushdown means the database applies your filters at the lowest level of data reading, so it never loads data it doesn't need. For a query returning 8K rows out of 367K, only those 8K rows cross the C++/Python boundary.

Download

DuckDB writes a gzip-compressed CSV to a temp file natively. The file is streamed to the client in 64 KB chunks, then deleted. At no point is the full CSV held in Python memory.

Concurrency

Background Tasks + SSE

CSV ingestion runs as a BackgroundTasks function. Starlette runs synchronous background tasks in a thread pool, keeping the event loop free. The SSE endpoint polls task state and emits events when status changes.

SSE (Server-Sent Events) is a built-in browser API where the server pushes updates to the client over a single HTTP connection. The browser's EventSource object handles reconnection automatically.
Data Integrity

Atomic Data Replacement

Each upload replaces the entire dataset atomically using a staging table:

1
Ingest CSV into movies_staging
2
BEGIN TRANSACTION
3
DROP TABLE IF EXISTS movies
4
ALTER TABLE movies_staging RENAME TO movies
5
COMMIT

Queries always see either the complete old dataset or the complete new one. If ingestion fails, the staging table is dropped and the original movies table is preserved.

Real-Time

SSE over WebSockets

Server-Sent Events was chosen over WebSockets because progress reporting is unidirectional (server to client). SSE uses standard HTTP with no upgrade handshake, and has native browser support via the EventSource API.

WebSockets create a two-way communication channel and require a protocol upgrade handshake. Since we only need to send updates from the server to the browser (not the other way around), SSE is simpler, lighter, and has better compatibility with proxies and load balancers.
API Design

Year vs Date Range

The CSV data only contains a year column, not a full date. The query API uses start_year / end_year as an honest interpretation of the "date range" requirement — rather than fabricating fake month/day values to match a date-based API.

Query Strategy

Genre Filtering: LIKE vs Normalization

Genres are stored as a single comma-separated string rather than normalized into a junction table:

  • DuckDB is an OLAP engine where denormalized data is idiomatic
  • ILIKE '%Action%' (case-insensitive) is pushed down to DuckDB's scan layer and is efficient on columnar data
  • A junction table would add join complexity with no performance benefit for analytical workloads
Substring matching could theoretically produce false positives (e.g., "Drama" matching "Melodrama"), but the dataset's genre vocabulary doesn't contain such conflicts.
State Management

Task State: In-Memory Dict

Task progress is tracked in a plain Python dict:

  • Single-process FastAPI — no cross-process state needed
  • Dict mutations are GIL-safe between the background thread and SSE generator Python's Global Interpreter Lock (GIL) ensures only one thread runs Python bytecode at a time. This makes simple dict read/write operations inherently thread-safe without needing explicit locks.
  • Task state is ephemeral by nature — it only needs to survive until the client reads it

Production upgrade: This would be replaced with Redis or a database-backed solution for horizontal scaling and persistence across restarts.
