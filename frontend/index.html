Design Decisions

Architecture & Design

Key design choices and the reasoning behind them.

DuckDB is an embedded OLAP (Online Analytical Processing) database — optimized for scanning and filtering large datasets, not transactional workloads.
  • Out-of-core processing — spills to disk when data exceeds RAM
  • Columnar storage — filter predicates pushed down to the scan layer
  • Native CSV reader — read_csv parses in C++, 170x faster than Python-side parsing
  • ACID transactions — enables atomic staging table swaps
  • Native gzip export — COPY TO COMPRESSION 'gzip' in a single C++ pass
The initial implementation used Python's csv.reader with executemany batching — 92 seconds for 367K rows. Switching to DuckDB's native read_csv reduced this to 0.5 seconds (170x faster).
  • TRY_CAST gracefully handles dirty data (e.g., "III" in year column becomes NULL)
  • strict_mode=false tolerates quoting irregularities
  • Header validation still uses stdlib csv.reader (just first line)
DuckDB handles CSV parsing, querying, and gzip export natively. Python's csv.reader is used only for header validation (one line). No heavy data processing libraries needed.
  • Upload — UploadFile streamed to a temp file in 1 MB chunks
  • Ingestion — DuckDB reads CSV directly from disk in C++
  • Query — DuckDB pushes WHERE filters to the scan layer
  • Download — DuckDB writes gzip to temp file; streamed to client in 64 KB chunks
Each upload replaces the dataset atomically using a staging table: ingest into movies_staging, then BEGIN; DROP movies; RENAME staging → movies; COMMIT;. Queries always see either the complete old dataset or the complete new one. If ingestion fails, the staging table is dropped and the original is preserved.
Server-Sent Events was chosen because progress reporting is one-way (server → client). No upgrade handshake, standard HTTP, native browser support via the EventSource API, and automatic reconnection built in.
Stack

Technologies

Each dependency chosen with intent. No bloat.

🐍

Python 3.13

Latest stable. Pinned <3.14 for pydantic-core compatibility. Native typing, asyncio, csv module.

⚡

FastAPI

Async-first, Pydantic validation, auto-generated OpenAPI docs. BackgroundTasks for offloading ingestion.

🦆

DuckDB

Embedded OLAP engine. Out-of-core processing, columnar storage, native CSV/gzip. Zero configuration.

✅

Pydantic

Type-safe request/response models. Automatic validation and serialization. First-class FastAPI integration.

📄

stdlib csv

Header validation only. Handles quoted fields with commas correctly. Zero dependencies, constant memory.

📡

Server-Sent Events

Real-time progress over standard HTTP. Native EventSource API. No WebSocket complexity for one-way data.

Security & Future

Measures taken and production-readiness improvements.

Implemented

  • 10 GB upload size limit (413 on exceed)
  • Query pagination (LIMIT/OFFSET, max 10K)
  • LIKE wildcard escaping in genre filter
  • Sanitized error messages (log internally)
  • XSS prevention (HTML escaping)
  • Path injection guard on temp files
  • Parameterized SQL queries

Future / Production

  • Redis-backed task queue (Celery/Dramatiq)
  • Rate limiting on upload endpoint
  • Authentication & API keys
  • Docker containerization
  • CI/CD with automated testing
  • OpenTelemetry instrumentation
  • Restrict CORS to specific origins
