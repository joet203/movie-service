FastAPI + DuckDB ~367K Rows Senior SWE Assessment
Context

What We're Building

Assessment for a Senior SWE role. Build a FastAPI movie database service that handles large datasets with minimal CPU/memory overhead. The provided movies.csv has ~367K rows (16MB) but the design must scale to much larger datasets.

Data shape: movie_name VARCHAR, year INTEGER (nullable), genres VARCHAR, rating DOUBLE (nullable). Genres are comma-separated strings. "Date range" is implemented as start_year/end_year since the data only has year.

Design philosophy: Senior-level restraint — no dependency bloat. Standard library + DuckDB's native API handles everything. No artificial delays or sleeps anywhere in the codebase.

Stack

Technology Choices

⚡ FastAPI + Pydantic 🦆 DuckDB 📜 Python stdlib 🐍 Python 3.13
  • FastAPI + Pydantic — provided template, async-native web framework
  • DuckDB — persistent on-disk analytical DB with (an OLAP engine — a database designed for reading and analyzing large data, not small transactions), (out-of-core processing — can handle data bigger than available RAM by spilling to disk), native read_csv for direct CSV ingestion
  • Python stdlib — csv.reader for header validation, tempfile for disk streaming
  • Python 3.13 — pinned >=3.13,<3.14 since pydantic-core doesn't support 3.14 yet

app/db.py

Database & Task State

DuckDB connection (module-level singleton):

  • _conn: duckdb.DuckDBConnection | None = None
  • init_db() — connects to movies.duckdb, runs:
CREATE TABLE IF NOT EXISTS movies (
    movie_name VARCHAR,
    year INTEGER,
    genres VARCHAR,
    rating DOUBLE
)
  • get_db() — returns _conn, raises RuntimeError if not initialized
  • close_db() — closes connection, sets _conn = None

Task state (module-level dict):

  • tasks: dict[str, dict] = {} — maps task_id to {"status", "progress", "error"}
  • Simple dict is fine — single-process FastAPI, no need for Redis/external store. Operations on Python dicts are (GIL-safe — Python's Global Interpreter Lock makes simple dict operations safe between threads).

app/movies.py

Route Handlers
GET /health
Return {"status": "ok"}
POST /datasets

Accept file: UploadFile, return TaskResponse with HTTP 202.

  1. Stream to temp file — read UploadFile in 1MB chunks, write to tempfile.NamedTemporaryFile(delete=False, suffix='.csv'). Never hold full file in memory.
  2. Generate task_id = str(uuid4())
  3. Store in tasks[task_id] with status "pending"
  4. Kick off background work via BackgroundTasks (FastAPI feature that runs work after the HTTP response is sent)
  5. Return TaskResponse(task_id=task_id)

Background: _ingest_csv(task_id, temp_path)

This was the initial plan, which used Python's csv.reader + DuckDB executemany for batch inserts with row-by-row progress tracking. The final implementation replaced all of that with DuckDB's native read_csv + TRY_CAST — much simpler and faster, as DuckDB handles CSV parsing in optimized C++ code instead of Python.

Runs as a synchronous function via BackgroundTasks (Starlette automatically runs sync functions in a thread pool, keeping the event loop free). No sleeps, no artificial delays.

  1. Set status to "processing"
  2. Validate header — quick first-line read with csv.reader, check column names match expected schema
  3. DuckDB native CSV read — CREATE TABLE movies_staging AS SELECT ... FROM read_csv(...) with TRY_CAST (attempts a type conversion and returns NULL on failure instead of erroring — gracefully handles dirty data like "III" in the year column)
  4. Atomic table swap (atomic swap — replacing data in one indivisible operation; readers see either all old or all new data, never a mix): BEGIN TRANSACTION; DROP TABLE IF EXISTS movies; ALTER TABLE movies_staging RENAME TO movies; COMMIT;
  5. Finalize — os.unlink(temp_path), set progress to 100%.
  6. Error handling — wrap in try/except, clean up staging table on failure, ensure temp file is removed in finally.
GET /tasks/{task_id}/events

SSE StreamingResponse for real-time progress updates.

  • Look up task_id in tasks dict — 404 if not found
  • Async generator yields data: {"status", "progress", "error"}\n\n
  • Polls tasks dict, yields on change, uses asyncio.sleep(0.5) as a non-blocking poll interval (not an artificial delay — standard SSE pattern to avoid busy-waiting; the async sleep yields control back to the event loop)
  • Stops when status is "completed" or "error"
GET /movies

Query by date range and genre(s), returning a list of movies.

  • Optional query params: start_year, end_year, genre
  • Validate start_year <= end_year if both provided — 400 otherwise
  • Build SQL dynamically with (parameterized queries — SQL with placeholders ? that prevent injection attacks)
  • Genre matching uses LIKE '%genre%' substring match — DuckDB applies (predicate pushdown — filters at the lowest data-reading level, avoiding loading unnecessary data)
SELECT movie_name, year, genres, rating
FROM movies
WHERE 1=1
  AND year >= ?   -- if start_year
  AND year <= ?   -- if end_year
  AND genres LIKE ?  -- if genre
GET /datasets/download

Download the full database as a gzipped CSV.

  1. Check table has data — 404 if empty
  2. Use DuckDB's native COPY ... TO ... COMPRESSION 'gzip' — the database handles gzip compression natively and efficiently
  3. Stream the .csv.gz file in 64KB chunks via StreamingResponse
  4. Set Content-Disposition: attachment; filename="movies.csv.gz"
  5. Clean up temp file after all chunks are yielded

DESIGN.md

Design Choices

Brief document covering:

  • Why DuckDB — out-of-core (can handle data bigger than available RAM by spilling to disk) analytical engine, zero-config, native read_csv for efficient ingestion
  • Why read_csv + TRY_CAST — DuckDB parses the CSV in optimized C++ code, TRY_CAST gracefully handles dirty data (NULLs instead of errors), no Python-side parsing needed
  • Memory management — streaming upload (1MB chunks), DuckDB-native CSV read (no Python-side row buffering), streaming download (DuckDB native gzip + chunked file read)
  • SSE design — dict-based task state, real progress from row counts, no artificial delays
  • Atomic data replacement — staging table + rename (no partial state visible to queries)
  • Trade-offs acknowledged — single-process task state (no horizontal scaling), task history not persisted across restarts, no granular progress (DuckDB handles ingestion as one operation)
