Movie Database Service

FastAPI + DuckDB — Architecture Plan & Requirements Coverage

FastAPI DuckDB Pydantic Python 3.13 stdlib csv.reader asyncio.to_thread uv

Upload endpoint accepts the CSV file and ingests it into DuckDB via a background task.

Endpoint

POST/datasets

Returns 202 Accepted with a task_id for progress tracking.

How It Works

1Stream UploadFile to a temp file in 1 MB chunks — the full file is never held in memory.
2Generate a UUID task_id, store it in the in-memory task tracker, return 202 immediately.
3Background task runs via asyncio.to_thread (non-blocking to the event loop).
4Fast line-count pass over the temp file for progress denominator (no parsing, just \n counting).
5Parse rows with stdlib csv.reader. Validate headers. Skip malformed rows gracefully.
6Batch insert via executemany into a staging table. Flush + update progress every 50K rows.
7Atomic swap: DROP movies; ALTER TABLE movies_staging RENAME TO movies; inside a transaction.
8Clean up temp file. Mark task as completed.
▶ Design Choices
Why DuckDB executemany over read_csv?

Batched executemany with parameterized INSERT flushes 50K rows at a time, keeping memory bounded. Using read_csv would be a single blocking call with no way to report intermediate progress.

Why stdlib csv.reader over Polars/pandas?

Zero extra dependencies. csv.reader correctly handles quoted fields (like comma-separated genres) and streams row-by-row. Polars/pandas would add significant dependency weight for no real benefit here.

Why staging table + atomic rename?

During ingestion, queries against movies continue to work with the old data. The swap is a single transaction — no partial state is ever visible to readers.

Trade-off: Two-pass file I/O (count then parse) doubles disk reads, but the first pass is a fast sequential scan with no parsing, and the OS page cache makes the second pass essentially free.
▶ Error Handling
  • Invalid CSV headers → task set to "error" with descriptive message
  • Rows with wrong column count → skipped (counted)
  • Unparseable year/rating values → skipped gracefully
  • Any exception → staging table dropped, temp file cleaned up, task marked as error
▶ Null Value Handling

Empty strings for year and rating are passed as None to DuckDB, stored as SQL NULL. Data fidelity is preserved — no coercion to 0 or default values.

year = int(row[1]) if row[1].strip() else None
rating = float(row[3]) if row[3].strip() else None

Endpoint

GET/datasets/download

How It Works

1Check table has data via SELECT COUNT(*). If empty → 404.
2DuckDB writes directly to a temp .csv.gz file:
COPY movies TO '<path>.csv.gz' (FORMAT CSV, HEADER, COMPRESSION 'gzip')
3Stream the compressed file to the client in 64 KB chunks via StreamingResponse.
4Temp file cleaned up in the generator's finally block after streaming completes.

Response Headers

  • Content-Type: application/gzip
  • Content-Disposition: attachment; filename="movies.csv.gz"
▶ Design Choices
Why DuckDB native COPY with COMPRESSION?

DuckDB handles CSV formatting and gzip compression in a single optimized pass. No Python-side row iteration, no manual gzip — the entire export is handled by DuckDB's native engine at C++ speed.

Why stream from temp file instead of piping?

DuckDB's COPY writes to a file path. By streaming that file in 64 KB chunks, we never load the full compressed output into memory. The temp file is deleted immediately after streaming.

Endpoint

GET/movies?start_year=&end_year=&genre=

All parameters are optional. Returns a JSON list of matching movies.

Parameters

  • start_year (int, optional) — inclusive lower bound on year
  • end_year (int, optional) — inclusive upper bound on year
  • genre (str, optional) — substring match against the genres field

Query Construction

SELECT movie_name, year, genres, rating
FROM movies
WHERE 1=1
  AND year >= $start_year   -- if provided
  AND year <= $end_year     -- if provided
  AND genres LIKE '%Action%' -- if provided

All filters use parameterized queries — no string interpolation, no SQL injection risk.

Validation

  • If both start_year and end_year provided: validates start_year ≤ end_year → 400 if not
  • No filters → returns all movies
▶ Design Choices
Why "year" instead of "date"?

The CSV data only has a year column, not a full date. Implementing start_year/end_year is the honest interpretation. This is documented in DESIGN.md.

Why LIKE for genre matching?

Genres are stored as a single comma-separated string (e.g., "Action, Crime, Drama"). LIKE '%Action%' provides simple substring matching without requiring schema normalization. DuckDB pushes the filter down to the scan layer for efficiency.

Why not normalize genres into a separate table?

DuckDB is an OLAP/columnar engine where denormalized data is idiomatic. A genre junction table would add join complexity for every query with no performance benefit — DuckDB's scan-layer LIKE filter is already efficient on columnar data. Normalization is an OLTP pattern that doesn't suit this use case.

Why single genre filter (not multi)?

Keeps the API surface simple and matches the original template's model. The LIKE pattern could be extended to multiple genres if needed, but for this assessment single-genre filtering is sufficient and clean.

The requirement: "Provide progress updates for long running requests > 2 seconds." This means: if a request takes longer than 2 seconds, the client should be able to get progress updates. Our async task + SSE pattern satisfies this.

Endpoint

GET/tasks/{task_id}/events

Server-Sent Events stream. Connect after receiving a task_id from POST /datasets.

How It Works

1POST /datasets returns 202 immediately with a task_id — the client is never left waiting.
2Client opens an SSE connection to /tasks/{task_id}/events.
3Server yields events with real progress percentages as the background task processes rows.
4Events include: status (pending/processing/completed/error), progress (0-100), error (if any).
5Stream ends when status reaches "completed" or "error".

Event Format

data: {"status": "processing", "progress": 45, "error": null}

data: {"status": "completed", "progress": 100, "error": null}
▶ Design Choices
Why SSE over WebSockets?

SSE is simpler, unidirectional (server → client), and built on standard HTTP. Perfect for progress reporting where the client only needs to receive updates. No handshake complexity.

Why real progress, not simulated?

The background task counts total rows first, then reports actual rows_done / total_rows. The progress percentage reflects real ingestion state — no estimation or faking.

Why in-memory task dict?

Single-process FastAPI — a dict is the simplest correct solution. No Redis or database needed. The dict is shared between the background thread and SSE generator via Python's GIL-safe dict operations.

Why asyncio.sleep(0.5) in the poll loop?

This is NOT an artificial delay. It's the standard SSE pattern to avoid busy-waiting. The async sleep yields control back to the event loop, allowing other requests to be served. Events are only emitted when progress has actually changed.

The service is graded on "overall performance and responsiveness, efficient management of CPU and memory."

Memory Management

  • Upload: Streamed to disk in 1 MB chunks. Never loaded into memory.
  • Ingestion: csv.reader streams row-by-row. executemany flushes every 50K rows. O(1) memory per row.
  • Queries: DuckDB pushes filters down to scan layer. Only matching rows are materialized.
  • Download: DuckDB writes compressed file natively. Streamed to client in 64 KB chunks.

CPU Efficiency

  • DuckDB executemany: Batched parameterized INSERT in 50K-row chunks for bounded-memory bulk insert.
  • asyncio.to_thread: Offloads blocking DuckDB/CSV work to a thread pool, keeping the event loop responsive.
  • Native gzip: DuckDB handles compression in C++ — no Python-side gzip overhead.

Responsiveness

  • Upload returns 202 immediately — no blocking on ingestion.
  • Event loop never blocked — all heavy work runs in background threads.
  • SSE provides real-time progress during long operations.
▶ Why DuckDB?
  • Out-of-core processing: DuckDB automatically spills to disk when data exceeds available RAM. Works with datasets far larger than memory.
  • Columnar storage: Analytical queries (filters, aggregations) are inherently faster with columnar layout.
  • Zero-config: Embedded database, no separate server process. Single file on disk.
  • ACID transactions: Enables the atomic staging table swap pattern.

The service is graded on "graceful error handling."

HTTP Error Responses

  • 400 — Invalid query params (e.g., start_year > end_year)
  • 404 — Unknown task_id in SSE endpoint
  • 404 — Download with no data in database

Ingestion Error Handling

  • Invalid CSV headers → task marked "error" with message, staging table dropped
  • Wrong column count in a row → row skipped
  • Unparseable year/rating → row skipped
  • Any unhandled exception → staging table dropped, temp file cleaned, task set to error
  • Old movies table preserved if ingestion fails — no data loss

Resource Cleanup

  • Temp files always cleaned up via finally blocks
  • Staging tables dropped on failure
  • DuckDB connection closed on app shutdown via lifespan handler

Improvements that would make sense for a production deployment but are out of scope for this assessment.

Data Quality

Genre Normalization

Currently genres are stored as a comma-separated string and matched with LIKE. A normalized junction table would enable exact genre matching (avoiding theoretical false positives like "Drama" matching "Melodrama") and support efficient multi-genre AND/OR queries. Skipped here because DuckDB is an OLAP engine where denormalized data is idiomatic.

Feature

Multi-Genre Filtering

Support querying by multiple genres simultaneously (e.g., ?genre=Action&genre=Drama) with AND/OR semantics. The current single-genre LIKE filter is simple and clean, but multi-genre support would be a natural extension.

Scale

Persistent Task State

Task progress is stored in an in-memory dict — lost on restart. For production, store task state in DuckDB itself or Redis. Would also enable horizontal scaling across multiple workers.

Scale

Concurrent Upload Handling

Currently, simultaneous uploads both try to swap the movies table. A queue or lock mechanism would serialize uploads, or each upload could target a versioned table with a pointer swap.

Performance

Query Pagination

The GET /movies endpoint returns all matching results. For very large result sets, adding limit/offset or cursor-based pagination would reduce response size and memory usage.

Performance

DuckDB Indexing

For frequently filtered columns like year, DuckDB's min-max indexes (zonemap) already help. Explicit indexing or partitioning by year could further improve scan performance at scale.

Feature

Sorting & Filtering by Rating

The API currently doesn't expose rating as a filter or sort parameter. Adding min_rating, sort_by=rating would be natural extensions.

Data Quality

Schema Flexibility

The ingestion pipeline validates against a fixed 4-column schema. Supporting configurable column mappings or auto-detecting CSV schemas would handle more varied data sources.
